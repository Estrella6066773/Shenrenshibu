# 词条编辑规范

本文继承并整理原 `正文/99-治理与模板/00-治理说明.md` 中的约定，作为 `试验` 架构下的**唯一规范入口**。若两处文字冲突，以本文件为准，并应及时回写 `正文/` 中旧版说明。

## 通用元数据（YAML 前置）

每条正式词条建议在文件首部使用下列字段（允许按项目工具微调键名，但须全库一致）：

```yaml
id: LORE-XXX-000
title: 词条标题
domain: entity | relation | rule | timeline | observation
type: 子类型代码
status: draft | review | frozen | deprecated
stability: low | medium | high
source: 官方档案 | 组织记录 | 民间流言 | 冲突证词
confidence: S | A | B | C | D
owner: 负责人标识或姓名
review_cycle: monthly | milestone
links:
  relates_to: []
  depends_on: []
  contradicts: []
visibility:
  author: full | partial | none
  role: full | partial | none
  player: full | partial | none
  system: full | partial | none
```

## 词条正文结构（Markdown）

建议由下列章节构成；可根据域省略不适用的节，但不得在缺少 `定义` 的情况下发布为 `review` 以上状态。

### 定义

- 写明本词条覆盖的边界与**不**覆盖的内容。
- 关键事实段后可接 `[投放注记]`（见下文）。

### 结构

- 组成要素、层级、与相邻词条的接口。

### 触发与变化（如适用）

- 何时变化、由何触发、变化后的结构或规则后果。

### 可观测后果

- 玩家或系统如何感知；宜与 `LORE-OBS-*` 或观测层草稿对齐。

### 争议与冲突

- 与哪些标识存在张力、当前裁决口径、待决问题。

## 投放注记块

在需要区分受众知情权的关键段落后，使用下列模板（可多段重复）：

```md
[投放注记]
- audience: 作者 | 角色 | 玩家 | 系统
- window: 阶段或事件或任务完成后可见
- channel: 档案 | 台词 | 新闻 | 任务反馈 等
- distortion: 无 | 轻度 | 显著（若显著，写明偏差方向）
```

注记只约束**可见性与口径**，不反向改写事实真值。若某段在各受众间无差异，可省略注记以保持简洁。

## 维护流程

1. **新增**：先占用标识并登记于 `ID注册表.md`，再写元数据与正文，最后补全 `relates_to` / `depends_on`。  
2. **修改**：必须检查依赖闭包；若影响对外包，同步更新 `40-玩家投放包/` 中对应版本与清单。  
3. **冻结**：`frozen` 后仅可通过评审单变更；须在 Git 或评审系统中保留说明。  
4. **弃用**：`deprecated` 后保留重定向说明，并给出替代标识。
